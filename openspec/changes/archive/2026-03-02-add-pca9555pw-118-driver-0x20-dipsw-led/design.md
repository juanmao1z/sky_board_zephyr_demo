## Context

当前工程已经在 I2C1 上接入了 INA226 与 AHT20，并建立了统一 `platform` 抽象层。PCA9555PW,118 需要作为新的 I/O 扩展器加入到同一总线，固定地址为 `0x20`。本次需求明确了端口语义：Port0 全部位作为拨码输入，Port1 全部位作为白色 LED 输出。设计目标是在不破坏现有服务架构的前提下，提供清晰可复用的平台接口。

## Goals / Non-Goals

**Goals:**

- 在板级 devicetree 中新增 PCA9555 设备节点并固定地址 `0x20`。
- 提供平台层接口以支持：
  - 读取 8 位拨码输入（`io0_0` ~ `io0_7`）。
  - 写入 8 位白色 LED 输出（`io1_0` ~ `io1_7`）。
- 驱动初始化时确保端口方向配置正确（Port0 输入，Port1 输出）。
- 异常路径保持安全输出（LED 默认关闭）并输出可诊断日志。
- 在 `app_Init()` 完成最小化初始化与状态日志，不阻塞主启动链路。

**Non-Goals:**

- 不实现中断模式输入采集，仅提供轮询读取接口。
- 不新增独立后台服务，不在本次接入中实现业务联动策略。
- 不扩展为通用 16-bit 任意方向动态配置接口（保持固定语义）。

## Decisions

### 1) 使用 Zephyr 原生 GPIO expander 驱动（PCA series）

- 决策: 在 DTS 使用 `compatible = "nxp,pca9555"`，由 Zephyr `GPIO_PCA_SERIES` 驱动接管。
- 原因: 减少自研寄存器驱动成本，保持与现有 Zephyr 生态一致。
- 备选:
  - 自行通过 I2C 读写寄存器实现: 可控更高，但维护与验证成本显著增加。

### 2) 平台接口固定为 DIP/LED 两个语义化操作

- 决策: 提供 `init/read_dipsw/set_leds` 风格接口，不暴露底层 GPIO 通用语义给业务层。
- 原因: 业务调用更直接，且与硬件连接关系一一对应，减少误用。
- 备选:
  - 暴露通用 pin read/write: 灵活但不符合当前固定板级用途，易出现方向误配。

### 3) 固定端口方向并在 init 阶段设置默认输出

- 决策: 初始化时统一设置 Port0 为输入、Port1 为输出，并将 LED 位图清零。
- 原因: 启动行为可预测，避免上电后 LED 随机亮灭。
- 备选:
  - 延迟到首次调用再配置: 简化 init，但会带来状态不确定窗口。

### 4) 失败策略采用“记录错误并降级继续”

- 决策: 初始化或读写失败只记录日志，不中断 `app_Init()`。
- 原因: 与项目当前外设接入策略一致，避免单器件异常影响全系统启动。
- 备选:
  - 失败即终止启动: 可靠性要求更强，但会降低系统可用性。

## Risks / Trade-offs

- [I2C 总线冲突] → 与现有 I2C1 设备地址或时序冲突可能导致访问异常; 通过固定 `0x20` 与上电日志验证缓解。
- [方向配置误用] → 若业务误当作双向口使用会导致行为不一致; 通过接口语义化和固定方向约束缓解。
- [极性认知偏差] → DIP 或 LED 板级电平极性可能与直觉相反; 通过文档明确位语义并在日志中打印原始位图缓解。
- [无中断模式] → 高频拨码变化无法事件驱动捕获; 当前需求允许轮询读取，后续再扩展中断。

## Migration Plan

1. 在 overlay 增加 PCA9555 节点与 alias。
2. 在 `prj.conf` 增加 GPIO expander 相关 Kconfig（如需显式使能）。
3. 新增平台抽象头文件和 Zephyr 实现文件。
4. 在 `app_Init()` 接入初始化与状态日志。
5. 执行 `west build` 验证，随后上板读取 DIP 与控制 LED 做基础联调。
6. 若出现回归，回滚新增 overlay 节点与平台接入即可恢复。
